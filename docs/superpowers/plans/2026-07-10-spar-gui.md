# spar GUI (PySide6 dashboard-pilot) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** `spar gui` — a native desktop dashboard-pilot (mockup variant A): live stream with per-side colors and filters, task board, dynamic gate panel with one-click decisions (consensus Accept auto-starts exec), run lifecycle controls (new debate / start exec / resume / stop). No embedded terminal — claude/codex stay in the user's terminal.

**Architecture:** One new package `spar/gui/` behind an optional dependency (`pip install spar-cli[gui]` → PySide6). The GUI is the FOURTH consumer of the existing agent-mode protocol — the engine is not modified: it reads `.spar/live.log` (reusing `spar.watch.follow` semantics), polls `spar.status.build_status()` in-process (import, no subprocess), and issues decisions by spawning the same CLI commands an agent would (`spar --continue --headless --quiet --gate …`). Gate buttons render EXCLUSIVELY from `pending_gate.options`, so an illegal decision is unrepresentable. Process ownership: the GUI is a solo pilot — it spawns and owns spar processes via `QProcess` (Stop = SIGINT → exit 130); coexistence with a driving agent is explicitly out of scope for v1 (documented; the GUI shows a read-only warning when it detects a running spar it did not start — lock held → "obserwuję tylko").

**Design tokens (from the accepted mockup):** app is committed-dark. Ground `#14171c`, panels `#1b1f26`/`#20252e`, line `#2c323c`, text `#d5dae2`/muted `#7d8590`; claude `#e8a854`, codex `#56c3d6`, spar-log `#9d8cff`, ok `#6fbf73`, warn `#e0b154`, gate `#e0679a`. Mono for stream (system mono stack), UI sans for chrome. One QSS file holds ALL colors as the single source.

**Tech Stack:** Python 3.11+, PySide6 (optional extra `gui`), pytest + pytest-qt (dev extra), `QT_QPA_PLATFORM=offscreen` for headless tests. Linux/macOS now; **Windows deferred by decision** (core uses fcntl — separate future tranche; noted in HANDOFF backlog).

## Global Constraints

- Engine untouched: NO changes under `spar/` outside `spar/gui/`, `spar/cli.py` (routing only) and `pyproject.toml` — with ONE sanctioned exception: `spar/status.py` may gain ADDITIVE fields (protocol evolution; see Task 4's `branches`). If anything else is missing from the protocol, STOP and surface it.
- `tests/conftest.py` sets `os.environ.setdefault("QT_QPA_PLATFORM", "offscreen")` BEFORE any Qt import, so the plain suite command stays `python3 -m pytest tests/ -q` headless (review #8).
- Core stays zero-dep: `import spar.gui` may require PySide6, but `spar` CLI without the extra must work exactly as today; `spar gui` without PySide6 prints an install hint and exits 2.
- Gate buttons only from `pending_gate.options`; consensus Accept default = accept + auto `spar exec --headless --quiet` (a secondary "Accept (tylko plan)" skips the auto-exec).
- final_merge NEVER auto-accepts.
- All subprocess spawns use the SAME python/venv (`sys.executable -m spar.cli` style or the resolved `spar` script) and always `--headless --quiet`.
- TDD with pytest-qt for logic-bearing pieces (view-models, gate mapping, runner state machine); pixel-level layout is smoke-checked, not asserted.
- Conventional commits; NO Co-Authored-By / AI-attribution trailers (hard rule).
- Suite green after every task: `python3 -m pytest tests/ -q` (439 passed, 2 skipped before this plan; GUI tests must skip cleanly when PySide6 is absent — `pytest.importorskip("PySide6")` at module top).
- README gets a GUI section in the SAME plan (memory rule: README ships with the feature).

### Task 2: process runner + toolbar (Opus)

**Files:**
- Create: `spar/gui/runner.py` (`SparRunner(QObject)`)
- Create: `spar/gui/toolbar.py` + new-debate dialog
- Test: `tests/test_gui_runner.py`

**Interfaces:**
- `SparRunner` — owns at most ONE `QProcess`; API: `start_debate(task_text, sides, first, tasks: bool)` (writes task_text to a temp task-file), `start_exec()`, `resume(gate_value: str | None, auto_exec: bool = False)` (debate vs exec chosen from `build_status()['phase']` — `None`/`"debate"` → `spar --continue`, else `spar exec --continue`; `auto_exec=True` is only legal for a consensus accept and chains `start_exec()` on exit 0), `resume_with_remarks(text: str)` (runner creates the temp remarks file — one remark per line — then delegates to `resume(f"remarks:{path}")`; the file is unlinked on `finished`), `stop()` (SIGINT). Signals: `started(cmd: str)`, `finished(exit_code: int)`, `state_changed(RunnerState)`.
- `RunnerState` enum drives toolbar enablement. PURE function `derive_state(alive, exit_code, status) -> RunnerState`, unit-tested exhaustively, covering the FULL exit-code table (review #7): alive+no gate→`RUNNING`; exit 10 (or not-alive+pending_gate)→`GATE_PENDING`; 130/4/crash→`RESUMABLE`; 5→`ABORTED` (Wznów on — resume re-pends the gate — and Nowa debata on); 0+phase done→`DONE`; 2→`ERROR` (stderr surfaced in the status bar; toolbar per underlying status); 3→`LOCKED` (read-only banner "inny proces spar trzyma lock — obserwuję tylko"; only Stop-less observation until the lock frees); fresh dir→`IDLE`.
- Stop (review #3): Qt's `terminate()`/`kill()` send SIGTERM/SIGKILL, NOT SIGINT — the runner sends POSIX SIGINT explicitly: `os.kill(process.processId(), signal.SIGINT)`, then waits; the engine's KeyboardInterrupt path returns 130 with state saved. POSIX-only by design (Windows deferred).
- Remarks temp-file lifecycle (review #4): the file passed as `--gate remarks:<path>` is opened by the CHILD process — it must outlive the spawn. Runner owns it: create under the GUI's tmp dir WITHOUT delete-on-close, remember the path, unlink it in the `finished` handler.
- Lock coexistence (review #2): `spar status` never touches the lock, so it CANNOT detect a running spar. The probe is a real non-blocking flock attempt on `.spar/lock` (open + `fcntl.flock(LOCK_EX|LOCK_NB)` + immediate unlock/close; POSIX-only, fine for v1). Held → `LOCKED` state. Spawns hitting exit 3 anyway surface the engine's message verbatim.
- Stale exec.json (review #6): `build_status` gives exec.json precedence forever, and a finished exec is never cleaned by the engine — so `start_debate()` on a status of `phase == "done"` FIRST archives the stale file (`exec.json` → `exec.json.prev-<epoch>`; GUI-owned, tested) so status/resume routing flips back to the debate. Without this, resume after a new debate would wrongly route to `spar exec --continue`.
- Auto-exec chain: the GatePanel's primary consensus button calls `resume("accept", auto_exec=True)`; "Accept (tylko plan)" calls `resume("accept")` (flag defaults False at the API level — the DEFAULT-ON behavior lives in the panel's primary button, not in the runner). Test: fake exits 0 → with auto_exec a second spawn (`spar exec --headless --quiet`) follows; without it, none.
- Commands spawned as `[sys.executable, "-m", "spar.cli", ...]`-equivalent with `setWorkingDirectory(project_dir)`; verify `python3 -m spar.cli --help` works — if not, fall back to the `spar` console script on PATH.

- [ ] **Step 1: failing tests** — `derive_state` truth table covering ALL rows above (incl. 2→ERROR, 3→LOCKED, 5→ABORTED). Runner integration with a FAKE spar script (tmp executable printing gate-pending line and exiting 10): start→signals, stop→child received SIGINT (fake traps it and writes a marker), resume builds the right argv with cwd=project_dir (fake records argv+cwd), `resume_with_remarks("line1\nline2")` creates a two-line temp file, passes `remarks:<path>` to the child (fake records it), the file exists while the child runs and is gone after `finished`, `start_debate` archives a stale done-phase exec.json, flock probe detects a held lock (helper process holds it).
- [ ] **Step 2–3: implement; PASS.**
- [ ] **Step 4: commit** — `feat(gui): SparRunner process pilot + toolbar state machine`
